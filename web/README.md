# MAI dashboard

The public face of the District Market Attractiveness Index built in this
repository. Next.js (App Router) + TypeScript + Firebase Web SDK v9, exported
as static files and served from Firebase Hosting. Every read happens in the
browser; there is no server.

```bash
npm install
cp .env.local.example .env.local     # fill in the Firebase web config
npm run dev                          # http://localhost:3000
npm run build                        # static export to out/
```

## Where the data comes from

**Firestore at runtime, once per session.** `src/lib/data.ts` fetches the whole
`mai_scores` collection (698 documents) plus `mai_runs` in one pass, normalises
it, and hands it to a React context. Every view — map, tables, charts, drawer —
derives from that single array. Nothing issues a query per district, because at
698 documents the whole collection costs less than the round trips would, and
hovering a map must never touch the network.

Two normalisation rules matter and both are in `src/lib/names.ts`:

- **District codes travel as both `"007"` and `"7"`** depending on which writer
  produced the file. Every lookup goes through `normaliseCode`, and the index
  registers both forms, so a padded and an unpadded key can never address
  different records.
- **Names are joined on an aggressively normalised key** — lowercase, diacritics
  stripped, `&` folded to `and`, punctuation removed, trailing administrative
  nouns dropped.

### Fallback

If `NEXT_PUBLIC_FIREBASE_*` is absent or the Firestore read fails, the app
reads the bundled snapshot in `public/data/` and says so in the status bar —
the dot turns amber and the label reads *bundled snapshot* instead of
*live · Firestore mai_scores*. Set `NEXT_PUBLIC_FORCE_LOCAL_DATA=true` to skip
Firestore deliberately for an offline demo.

The snapshot is generated by `scripts/build_fallback.py`, which builds district
documents through `mai/publish.py::build_documents()` — the same function the
publisher uses — so the fallback and the live read carry an identical shape and
the UI has exactly one contract to satisfy.

```bash
python3 scripts/build_fallback.py     # or: npm run data
```

### Artefacts that are not in Firestore

The validation table, benchmark pack, ML falsification, coverage report,
imputation sensitivity, crosswalk review and GenAI evaluations are analysis
outputs, not published documents. They live only in
`analysis/audit/_cache/v2/` and are always read from `public/data/`. Which
artefact surfaces where:

| artefact | page |
|---|---|
| `scores_v2_with_intervals.csv`, `narratives_v2.json` → `districts.json` | everywhere; the map, `/districts`, the drawer |
| `run_v2.json` | status bar, `/methodology`, `/data-quality` |
| `validation_v2.csv` | `/validation` |
| `benchmarks_v2.json` | `/validation` |
| `ml_validation_v2.json` | `/validation` |
| `coverage_report_v2.csv` | `/data-quality` |
| `crosswalk_review_v2.csv` | `/data-quality` |
| `imputation_flags_v2.json` | the district drawer |
| `imputation_spearman_v2.csv` | `/data-quality` |

Not surfaced, deliberately: `features_imputed_v2.csv`, `provenance_v2.csv`,

with no reader) or a redundant projection of `districts.json`. The crosswalk

which is the merged view of all three.

## Map assets

`public/` ships `india_states.geojson` (36 features, `NAME_1`) and
`india_crown.geojson` (the dashed disputed-territory outline). District
polygons are not in the repo by default and are produced by three scripts, in
this order:

```bash
python3 scripts/build_geo.py        # download ADM2, attach states, simplify, split
python3 scripts/build_name_map.py   # join shapeName -> district code
python3 scripts/fix_winding.py      # rewind rings for d3-geo
```

**Provenance.** geoBoundaries `gbOpen IND ADM2`, resolved through
`https://www.geoboundaries.org/api/current/gbOpen/IND/ADM2/` — 735 features
with `shapeName` / `shapeID`, no state property. The raw download (~48 MB) is
cached outside the repo.

**Simplification.** mapshaper 0.7.46:

```
npx mapshaper <src> -simplify visvalingam weighted 4.0% keep-shapes \
  -o precision=0.0001 force public/india_adm2.geojson      # 1.69 MB, all districts
npx mapshaper <src> -simplify visvalingam weighted 18% keep-shapes \
  -o precision=0.0001 force public/districts/<slug>.geojson # per state, ≤707 KB
```

`public/districts/index.json` maps each state slug to its file and district
count, so a state drill-down can load only its own districts.

**States are attached geometrically** — each district's interior point is tested
against the state polygons. Two needed a nearest-centroid fallback
(`Lakshadweep`, and a placeholder polygon since removed), and a handful of
coastal enclaves (Diu, Mahe, Puducherry, Phek, Nandurbar) land in a neighbour's
polygon and are corrected by the explicit `STATE_FIX` table in
`scripts/build_name_map.py`.

### Two traps worth knowing about

**Ring winding.** d3-geo clips on the *sphere*, and its convention is the
inverse of RFC 7946: it wants exterior rings wound **clockwise** and reads a
counter-clockwise one as *the whole globe except this shape*. geoBoundaries
ships RFC-compliant counter-clockwise rings, so straight from mapshaper every
district projected to identical sphere-sized bounds and the last one drawn
painted the entire map white — while the DOM showed 691 correctly coloured
paths and every join was fine. `scripts/fix_winding.py` rewinds them; it is
idempotent and settles the direction by measuring `india_states.geojson`, the
layer already known to render correctly.

**Placeholder polygons.** The source contains a `DATA NOT AVAILABLE` feature of
22.9 square degrees — larger than any real district — spanning most of northern
India. `build_name_map.py` strips it from the published files.

### Name matching

Alias table first, then strict within-state exact, then within-state fuzzy
(difflib ≥ 0.86, greedy and one-to-one). There is deliberately **no national
fallback**: an earlier version had one and it silently matched Sikkim's compass
districts to Delhi's, Karnataka's Bijapur to Chhattisgarh's, and Maharashtra's
Raigarh to Chhattisgarh's. A district joined to the wrong state is an invisible
error; a white polygon is a visible one.

Current result: **696 of 734 polygons matched** (47 of them fuzzy, all listed
for audit), **38 polygons unmatched**, **2 scored records with no polygon**.
The unmatched are almost entirely districts created after the 2011 Census
spine — Manipur's and Mizoram's new districts, Tamil Nadu's 2019 splits,
Alipurduar, Jhargram, Kalimpong, Palghar — which have no Census population, no
SECC row and no NFHS sample, so there is nothing to score them from. They
render white with a "no data" tooltip, are logged to the browser console, and
are listed in full on `/data-quality`.

## Environment

Copy `.env.local.example` to `.env.local`. Every variable is a **public client
identifier** — Firebase web config is not secret. The service-account JSON in
the repo-root `.env` is server-only and must never appear in a `NEXT_PUBLIC_*`
variable or in this directory.

| variable | purpose |
|---|---|
| `NEXT_PUBLIC_FIREBASE_API_KEY` | web API key |
| `NEXT_PUBLIC_FIREBASE_AUTH_DOMAIN` | `<project>.firebaseapp.com` |
| `NEXT_PUBLIC_FIREBASE_PROJECT_ID` | project id |
| `NEXT_PUBLIC_FIREBASE_STORAGE_BUCKET` | storage bucket |
| `NEXT_PUBLIC_FIREBASE_MESSAGING_SENDER_ID` | sender id |
| `NEXT_PUBLIC_FIREBASE_APP_ID` | app id |
| `NEXT_PUBLIC_FORCE_LOCAL_DATA` | `true` to skip Firestore entirely |

Firestore rules must permit public reads of `mai_scores`, `mai_runs`,
`districts` and `pipeline_status` for the site to read them anonymously.

## Deploy

The app is a Next.js **static export** — `next build` writes plain files to
`out/` and there is no server runtime, so it hosts anywhere that serves static
files.

### Netlify (recommended)

`netlify.toml` at the repo root configures everything: it points Netlify into
`web/`, runs `npm run build`, publishes `web/out/`, and mirrors the cache
headers below. Because the export is fully static, the `@netlify/plugin-nextjs`
runtime is deliberately **not** used.

1. Connect the GitHub repo in Netlify — the root `netlify.toml` is picked up
   automatically (`base = "web"`), so no build settings need to be typed in.
2. In **Site configuration → Environment variables**, add the
   `NEXT_PUBLIC_FIREBASE_*` values (public client identifiers, same names as
   `.env.local.example`). Without them the site still deploys and falls back to
   the bundled snapshot in `public/data`; set `NEXT_PUBLIC_FORCE_LOCAL_DATA=true`
   to force that offline mode.
3. Deploy. Every route is prerendered to its own HTML file, so no SPA rewrite is
   needed and `out/404.html` is served automatically for unknown paths.

For a one-off manual deploy without Git:

```bash
npm run build
npx netlify-cli deploy --dir=out --prod
```

### Firebase Hosting (alternative)

```bash
npm run build
npx firebase-tools deploy --only hosting     # after setting the project in .firebaserc
```

Both `netlify.toml` and `firebase.json` serve `out/` with immutable caching on
`_next/static` and a one-hour revalidate on the GeoJSON and data payloads.

## Notes on the design

Dark instrument-panel shell: `#0B0F14` page, `#141B23` panels, `#1F2A35`
borders, `#3FB6A8` accent used **only** for interaction and selection. The
choropleth ramp (`#1B2A38 → #F2C14E`) is deliberately a different family from
the accent so that a high score never reads as a selected state; selection is a
`#3FB6A8` stroke and never a fill change. Space Grotesk for headings, IBM Plex
Mono with tabular figures for every number so table columns align and values do
not jitter on hover.

Shading is by **percentile within the distribution**, not by raw score. A
bounded composite with most of its mass in the middle would otherwise render
the whole country in two indistinguishable shades, and where a district sits
relative to the others is what a relative index actually measures.

Charts are hand-rolled SVG. A charting library brings its own type scale,
palette and tooltip, and the point of this shell is that one palette and one
type system hold across map, tables and charts.

Below `lg` the map is replaced by a pointer to the searchable district table
rather than left as an unusable pinch-zoom target.
